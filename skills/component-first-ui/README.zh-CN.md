# Component-First UI

[English](README.md)

`component-first-ui` 指导 Codex 基于项目维护的 UI 知识库完成前端实现和审查。它从组件目录中发现可复用能力，遵循项目技术栈和可访问性约束，并在 UI 系统发生持久变化时同步更新相关文档。

权威 agent 工作流位于 [SKILL.md](SKILL.md)。

## 适用场景

- 实现页面、响应式布局或可复用组件；
- 新增或审查交互、动效、反馈、表单、导航或数据展示；
- 在项目组件、批准的组件库和原生平台能力之间选择实现方式；
- 为尚未建立 UI 开发文档的项目初始化文档；
- 审查 UI 修改中的组件复用、可访问性、文档漂移和针对性验证。

它不是通用的视觉风格预设，而是遵循目标仓库已经记录的 UI 系统和经过验证的源代码事实。

## 目标项目必需的入口

该工作流要求目标项目提供 `docs/ui-development.md`。此文档负责把 Codex 路由到相关组件目录、约束、技术栈证据和验证命令。

如果缺少该入口，skill 会执行一次面向 UI 的扫描，并至少创建：

```text
docs/
  ui-development.md
  ui/
    components/
      index.md
```

当可复用组件大约不超过 30 个时，使用紧凑目录。更大的系统按类别或功能域拆分，但导航层级不超过两层。

## 工作流概览

1. 阅读仓库指令和 `docs/ui-development.md`。
2. 仅在必需入口缺失或整体失效时初始化 UI 文档。
3. 将请求转换为语义能力，并搜索已经记录的组件目录。
4. 按顺序选择复用、组合、扩展、从批准的组件库新增或本地实现。
5. 遵循项目的组件边界、样式、图标、动效、响应式行为和可访问性规则。
6. 运行与改动风险相称的检查，并检查相关交互状态。
7. 当持久能力或约束发生变化时，更新负责该内容的 UI 文档或目录。

## 示例提示

```text
使用 $component-first-ui 基于项目现有组件实现响应式账户设置页面。

使用 $component-first-ui 审查这个对话框的复用方式、键盘操作、焦点管理和文档漂移。

使用 $component-first-ui 添加缺失的可复用空状态模式，不要引入新依赖。
```

## 实现方式优先级

| 优先级 | 实现方式 |
| --- | --- |
| 1 | 复用已经记录的项目组件。 |
| 2 | 组合已经记录的基础组件和模式。 |
| 3 | 从项目批准的 UI 库中添加缺失组件。 |
| 4 | 仅在策略允许且存在明确能力缺口时选择兼容依赖。 |
| 5 | 当依赖不必要或被禁止时，使用语义化原生元素和项目代码。 |

复杂控件需要按风险检查键盘操作、焦点、ARIA、层级、滚动、关闭行为、响应式表现、触控和减少动态效果设置。

## 预期交付说明

该 skill 会报告实现或审查的 UI、复用或新增的组件、依赖变化、文档更新以及实际执行的检查。它不会把未运行的视觉、浏览器、测试或构建检查描述为已经通过。

## 维护说明

- 保持入口文档精简，将详细信息通过链接路由到其他文档。
- 每个可复用组件只能有一个权威目录记录。
- 不要记录页面私有片段、临时实验、生成的内部代码或源类型中已经完整表达的 props API。
- 局部修复孤立的文档漂移；除非大部分文档模型已经失效，否则只重新扫描受影响的 UI 区域。
